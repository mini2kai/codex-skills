# Work Orchestrator

## 定位

本 Skill 是手动触发的轻量工作模式，用于在正式实施前完成分析、取证、方案设计、验证计划和专业 Skill 编排。

它不是固定流水线，也不是替代专业 Skill 的总包实现器。它只负责让工作先变清楚：目标是什么、证据在哪里、影响哪些范围、推荐怎么做、如何验证、是否需要进入实施。

各专业 Skill 仍然可以独立使用。只有当用户明确启用总控编排时，本 Skill 才按当前任务需要选择一个或多个专业 Skill 协助分析。

## 触发规则

推荐通过 `$work-orchestrator` 或 UI 手动选择来启用本 Skill。启用后，按以下意图进入总控编排：

- 使用 `$work-orchestrator`
- 启用总控编排
- 编排分析
- 先分析不修改
- 先定位再出方案
- 先给方案不要改代码
- 先评估影响范围
- 先出验证方案

普通的“帮我改”“实现一下”“修复这个问题”“加一个字段”不应自动触发本 Skill。若未显式启用本 Skill，优先按普通开发或对应专业 Skill 处理。

如果用户明确指定某个专业 Skill，优先按该专业 Skill 独立处理，不要强制插入本 Skill。若该 Skill 在当前环境不可用，明确说明不可用并给出替代分析方式。

## 职责边界

本 Skill 阶段只做：

- 理解用户目标和当前现象
- 判断请求类型和复杂度
- 阅读代码、配置和调用链
- 设计数据库只读验证 SQL
- 设计日志检索条件，或编排只读日志 Skill 取证
- 分析影响范围、风险点和依赖关系
- 输出推荐方案、实施步骤和验证计划
- 判断是否需要调用其他专业 Skill

本 Skill 阶段不做：

- 修改业务代码
- 修改配置
- 写数据库
- 修改线上配置
- 删除、移动、清理文件
- commit、push、部署

方案输出后，如果用户明确回复“按方案处理”“授权修改”“继续实现”，再进入实施阶段。实施阶段可以使用普通 Codex 流程或对应专业 Skill，并遵守对应 Skill 的安全规则。

## 工作方式

先用最小必要步骤推进，不要机械套模板。

1. 先判断任务属于问题定位、新需求分析、数据验证、日志排查、配置/page_id、导出、异步任务、外部系统集成，还是多类混合。
2. 再判断证据来源：代码、数据库、日志、配置、页面、接口、历史上下文。
3. 能从仓库和上下文获得的信息优先自行查证；缺少关键条件时再向用户提问。
4. 只收集足以支持当前判断的证据，不展开无关模块。
5. 每个确定性结论都要能追溯到证据；证据不足时标注为假设或待验证。
6. 最后给出可执行方案和验证方式，并停在实施前确认。

## 输出原则

不要固定输出完整大模板。根据任务复杂度自适应：

- 简单问题：给关键判断、依据、下一步。
- 中等问题：给现象、证据、原因、方案、验证方式。
- 复杂问题：再展开影响范围、风险点、实施步骤、回归范围。
- 缺少证据：明确写出“当前是假设”或“待验证项”。

只有在确实涉及代码、数据库、日志时，才输出对应证据段。不要为了模板完整而填充无关内容。

## 证据规则

代码证据需要包含：

- 文件路径
- 行号
- 关键逻辑
- 该逻辑如何支持当前判断

数据库证据需要包含：

- 表名和字段
- 查询条件
- 只读验证 SQL
- 预期结果或如何解释结果

日志证据需要包含：

- 日志来源
- 时间范围
- 关键字或 trace id、request id、user id、page id、task id 等标识
- 关键日志行
- 日志如何支持当前判断

如果无法直接读取数据库或日志，输出可执行的只读 SQL、日志检索条件和预期判断标准。

## 专业 Skill 独立性

各专业 Skill 必须保持独立可用：

- 不把本 Skill 作为其他 Skill 的前置步骤。
- 不复制或覆盖专业 Skill 的安全规则、脚本规则和领域细节。
- 用户明确点名某个专业 Skill 时，直接使用该专业 Skill，不强行套总控编排。
- 本 Skill 只在被手动启用时参与，并只负责判断“需要哪些能力”。
- 进入专业 Skill 后，遵守该专业 Skill 自己的流程和风险边界。

## 专业 Skill 编排

本 Skill 按“能力需求”动态编排，不按固定任务模板编排，也不假设某台机器一定安装了某个专业 Skill。一个请求可以不调用专业 Skill，也可以调用一个或多个当前环境可用的专业 Skill。

编排时按以下顺序判断：

1. 先从当前会话可见的 Skill 列表中读取 `name` 和 `description`。
2. 再根据用户目标判断缺少哪些能力，例如代码分析、数据库只读验证、日志读取、页面配置、文档同步、变更文档等。
3. 只选择当前环境确实可用、且 `description` 与目标匹配的 Skill。
4. 如果没有合适的专业 Skill，不要假装可以调用；改为自己完成通用分析，或明确说明缺少对应专业 Skill。
5. 如果用户明确指定某个专业 Skill 但当前环境不可用，说明该 Skill 不在当前可用列表中，并给出可替代的分析步骤。

调用专业 Skill 时，先说明三点：

- 为什么需要这个 Skill。
- 需要它补充什么证据、分析或产物。
- 它的结果如何回到当前总控分析中。

不要为了完整性调用无关 Skill；也不要因为总控已启用，就阻止专业 Skill 独立完成它擅长的工作。后续新增或减少专业 Skill 时，只根据当前可用 Skill 的 `description` 和用户目标做编排。

## 授权交接

当已经形成方案但尚未实施时，用简短方式停住并确认：

```text
当前 work-orchestrator 阶段已完成分析和方案设计，尚未修改代码或配置。
如需继续实施，请明确回复：按方案处理 / 授权修改 / 继续实现。
```

如果方案涉及修改，交接前列出：

- 预计修改的文件、模块或配置
- 每处修改的原因
- 主要风险点
- 验证步骤

用户授权后，不再把本 Skill 当作限制实施的理由；应进入实施流程并按实际任务选择普通开发方式或专业 Skill。
