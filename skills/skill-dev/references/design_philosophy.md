# Skill 设计理念

## 核心原则

**脚本强制执行的，留。给 AI 看的说明文字，删。**

一个 skill 的价值不在于教 AI 怎么做事，而在于用代码定义 AI 不能越过的边界。边界以内，完全信任模型的能力。

## 围栏模型

每个 skill 分为两层：

| 层 | 实现方式 | 示例 |
|---|---|---|
| 围栏（硬约束） | 代码 `raise`/`exit`/硬上限 | SQL 只读检查、行数上限、超时截断、凭据脱敏 |
| 围栏内（自由区） | AI 自行决定 | 写什么 SQL、怎么沟通、怎么解释结果 |

围栏由代码保证，AI 无法选择是否遵守。围栏内的事不做任何规定。

## 什么该留

- **代码级的安全检查**：白名单、黑名单、输入验证、硬上限
- **凭据保护逻辑**：脱敏函数、环境变量引用、不落盘策略
- **审计留痕**：操作日志、拦截记录
- **测试**：证明围栏真的有效
- **DESIGN.md**：给人看的设计理念，随 skill 更新

## 什么该删

- **教 AI 怎么思考的流程**：模型会自己判断
- **话术模板**：模型会说话
- **常见命令/SQL 模板**：模型知道怎么写
- **错误处理指南**：脚本输出的 JSON 已自描述
- **平台绑定配置**（openai.yaml 等）：会过时
- **驱动安装文档**：模型知道怎么装

判断标准：**如果把这段内容删掉，模型能不能自己完成？** 能就删。

## SKILL.md 应该怎么写

SKILL.md 只写三件事：

1. **围栏规则**：代码强制了什么，不可绕过
2. **脚本入口**：怎么调用
3. **围栏内自由**：明确声明 AI 在边界内自行决定

不写：触发判断逻辑、分步流程、引导话术、错误分支处理。

目标行数：40-60 行。超过 100 行说明在教 AI 做事。

## DESIGN.md

每个 skill 应有一个 `DESIGN.md`，给人看（不是给 AI 看）：

- 这个 skill 解决什么问题
- 设计理念（为什么这样做）
- 实现思路（围栏怎么建的）
- 什么被刻意删掉了，为什么
- 文件结构和职责划分

DESIGN.md 随 skill 更新同步维护。它是你的设计资产，也是对外输出的素材。

## 可复用性检查

开发或优化 skill 时问自己：

1. 这段代码换一个 AI 平台还能用吗？→ 能用才是真资产
2. 这段规则两年后模型更强了还需要吗？→ 围栏需要，流程不需要
3. 这个文件删掉后 skill 还能正常工作吗？→ 能正常工作就删

## 流程规则必须代码化

如果一件事"必须做"但只写在文档里，它一定会被遗漏。

判断标准：**这条规则被跳过时会导致问题吗？** 会就必须用代码强制。

示例：
- "改了 skill 要递增 manifest 版本" → `skill_preflight.py` 检查 git diff 与版本是否匹配
- "提交前必须通过测试" → 验证脚本在流程中自动运行
- "SQL 必须只读" → `sql_guard.py` 直接 raise

反面示例（不需要代码化的）：
- "commit message 用中文" → 忘了也不会造成损害
- "DESIGN.md 要随 skill 更新" → 提醒即可，不影响功能

**规则：任何出现过遗漏且导致实际问题的流程步骤，都必须在下一次迭代中变成代码检查。** 不靠记忆，不靠自觉，靠脚本拦截。

## 围栏 vs 提示词：如何判断

遇到一条安全规则时，问：**如果 AI 忽略了这条规则，会怎样？**

- 如果后果严重（数据丢失、凭据泄露、不可逆操作）→ 必须是围栏（代码强制）
- 如果后果可控（输出格式不对、多查了一次）→ 可以交给模型自觉

围栏的本质是：**你不信任任何调用方（包括未来更强的模型）时，仍然能保证安全。**

提示词适合的场景：触发路由、输出偏好、交互风格——这些即使做错了也不会造成损害。

## 持续迭代原则

随着模型能力提升，skill 应该越来越薄：

- 每次优化时审视：哪些流程描述已经不需要了？删。
- 围栏不会因模型进步而贬值——即使 AGI 来了，你也不想让它 force push
- 新增围栏规则时必须同步加测试
- DESIGN.md 记录每次重大决策的理由，方便回溯

**方向：skill 随时间变短，围栏代码变厚、测试变多。**

## 推送前文档同步规则

每次修改推送前，必须根据修改内容同步更新所有相关文档：

- 涉及 skill 的改动 → 更新该 skill 的 `DESIGN.md` / `DESIGN_cn.md`
- 涉及仓库级变更 → 更新 `docs/` 下相关文件
- 新增围栏 → 更新 SKILL.md 围栏列表
- 版本递增 → manifest.json（已有代码检查）

不允许"代码改了但文档没跟上"的状态被推送到远端。

## 文档双语规则

面向外部的 md 文档（README、INSTALLATION、docs/、DESIGN）提供中英双版：

- 英文为主文件（`.md`），中文为 `_cn.md`
- 主文件顶部加 `[中文版](./xxx_cn.md)` 链接
- SKILL.md 和 references/ 是内部文件，保持中文单版不动

## 仓库作为实践产物

这个仓库不是一个产品——是在真实研发场景中实践 AI Agent 安全协作的产物。

- 每个 skill 是一次具体问题的解决方案
- DESIGN.md 是解决过程中的思考记录
- docs/ 是从实践中提炼的方法论
- 仓库结构本身演示了围栏模型的落地方式

对外输出时，重点不是"这个仓库有几个 skill"，而是"AI Agent 协作中有这些风险，我用代码围栏这个方法解决了"。

## 事故案例库

`references/incidents/` 目录持续记录真实事故和经验教训。

每次遇到"规则被跳过且造成实际问题"时：
1. 记录事故（发生了什么、根因、影响、经验）
2. 判断能否围栏化
3. 能则加代码检查，不能则强化提示词规则并标注为"已知薄弱点"

事故是围栏进化的燃料。

## 参考案例

`postgres-query` 是按此理念重构的第一个 skill：

- `sql_guard.py`：零依赖 SQL 安全检查器，任何项目可用
- `pg_common.py`：连接管理 + 脱敏 + 审计
- `test_sql_safety.py`：42 个测试用例证明围栏有效
- SKILL.md：40 行，只有围栏规则和入口
- DESIGN.md：设计理念和实现思路的完整记录
