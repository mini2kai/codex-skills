# 原则

这 5 条原则是我从真实 AI 协作场景中提炼出来的。它们不依赖任何特定平台或模型——无论你用 Codex、Claude、GPT 还是未来的什么 Agent，这些约束逻辑都适用。

---

## 1. 默认只读，写操作需要显式授权

AI Agent 的默认动作应该是"看"而不是"改"。

读代码、读日志、读数据库、读文档——这些随时可以做。但写代码、改配置、执行 SQL、push 代码——每一个都需要人明确说"可以"。

不是因为 AI 会做错。是因为即使做对了，如果不是你想要的，撤回的成本远大于确认的成本。

**实践方式：**
- 数据库连接默认只允许 SELECT/WITH/SHOW/EXPLAIN
- DDL/DML 只生成 SQL 文本，不执行
- Git 操作中 commit 和 push 分开确认
- 文件修改前先展示 diff 或方案

---

## 2. 高风险操作不是禁止，是加确认门槛

"禁止 AI 做 X"是最简单的安全策略，但也最粗暴。

真正有用的模型是：AI 可以做任何事，但危险程度越高，确认门槛越高。日常操作静默通过，高风险操作需要人看一眼再放行。

门槛不是摩擦——是给人一个介入点。

**风险分级示例：**
- 无门槛：读代码、读日志、读文档、生成方案
- 轻确认：修改代码文件、生成 commit
- 重确认：push 到远程、执行数据库写入、合并分支
- 禁止：force push、reset --hard、删除分支/tag、直连生产写数据

---

## 3. AI 应该先证明它理解了问题，再被允许动手

大多数 AI 协作事故不是"AI 写错了代码"，而是"AI 解决了一个不存在的问题"。

要求 AI 先输出它的理解——问题是什么、根因假设是什么、打算改什么、不改什么——然后人确认理解正确后，再授权实施。

这不会拖慢节奏。一个理解正确的 AI 一次就能做对；一个理解错误的 AI 做三轮也是错的。

**实践方式：**
- 编排模式中设置阶段门：Intake → Evidence → Plan → (授权) → Execute → Verify
- Plan 阶段结束必须停住，输出方案，等人确认
- 证据不充分时不给确定结论，标注"待验证"
- 方案中列出"不改什么"和"改什么"同样重要

---

## 4. 安全规则跟着任务走，不是全局开关

"AI 不能访问服务器"是全局策略。但真实需求是"AI 可以查日志，但不能执行命令"。

把安全规则绑在任务粒度上，而不是全局一刀切。不同的任务有不同的信任边界，同一个 AI 在不同上下文里应该有不同的权限。

**实践方式：**
- 服务器日志：只允许通过白名单脚本读取，不给 SSH shell
- 数据库：按 profile 区分权限，生产 profile 永远只读
- Git：日常开发可以 commit，但 push 到 main 需要额外确认
- 文件系统：skill 自带的脚本可以执行，任意命令不行

---

## 5. 凭据和本地状态永远不进 AI 的持久上下文

AI Agent 的对话历史、内存、日志、输出文件——任何可能被持久化的地方，都不应该出现：

- 数据库连接串、密码
- SSH 密钥、token、cookie
- 生产环境地址和凭据
- 用户的 Excel 数据、客户信息

临时使用可以——传一个连接串让 AI 查一次数据没问题。但这个信息不应该被记住、被写入文件、被提交到 Git、或者出现在下一次对话中。

**实践方式：**
- 本地配置文件用 `.local.json` 后缀，.gitignore 里排除
- 密码通过环境变量引用（passwordEnv），不明文写入配置
- AI 生成的 Excel/日志/preview 文件不提交 Git
- 临时连接信息用完即弃，不写入持久 profile
